Building a Command-Based, Self-Owned Desktop
Automation Platform That Evolves Into an
LLMAssisted Office Agent

Executive summary

A "self-owned" desktop automation platform (no thirdparty agent frameworks) is best built as a command
runner + workflow engine first, then progressively augmented with state observation, validation/
recovery, evidence memory, and finally an LLM orchestration layer that converts natural language into
structured actions and handles edge cases. The key design decision is to treat the LLM as one module
(planner/interpreter) inside a deterministic automation system--rather than letting the LLM directly drive
every click.

For browser tasks, Playwright is a strong default because it provides robust, crossbrowser automation and
supports both sync and async APIs in Python, including isolated "browser contexts" that can avoid writing
session state to disk when desired. 1 For Windows desktop apps, pywinauto is a practical default
because it is designed specifically to automate the Windows GUI via supported accessibility/backends and
includes patterns for attaching to apps and finding controls. 2 For OCR and document/image
understanding, an effective starting stack is OpenCV preprocessing + Tesseract OCR, with
humanintheloop review when confidence is low; Tesseract is an opensource OCR engine with an API and
a documented training workflow. 3

To evolve into an LLMassisted office agent, use the OpenAI Responses API for multimodal (text+image)
inputs, stateful interactions, and tool/function calling. 4 Enforce structured action outputs using
Structured Outputs (JSON Schema constrained outputs) to reduce syntax errors and keep the LLM "inside
the rails." 5 For safety, implement a permission model and require confirmations for highimpact actions;
OpenAI's "computer use" guidance also emphasizes humanintheloop for high impact actions and treating
page content as untrusted input. 6

Reference architecture and data flow

Core principle

Build an automation platform that is deterministic by default, observable, and recoverable; the LLM
layer becomes an optional "brain" that proposes actions but does not bypass your platform's safety,
validation, and logging.

Module map and responsibilities

Command UI - A desktop app that accepts typed commands ("run daily report"), lets you pick saved
workflows, and shows run status, logs, and exceptions.

                                                                           1
Command registry - A catalog of commands with metadata: description, required parameters, associated
workflow(s), allowed apps/domains, and risk level.

Workflow engine - Executes workflows as a state machine: step  validate  next step; supports
branching, retries, and resume-from-failure.

Observer / state reader - Creates a structured snapshot of "what's happening now" (active window, UI tree
where available, browser URL/DOM state, known popups, optional screenshot). - Prefer eventdriven
observation when possible: Windows UI Automation allows subscribing to events to reduce polling (better
performance). 7

Executor - Low-level action runner: click, type, hotkeys, open app, navigate browser, handle files. - Desktop:
pywinauto (Windows-first) 8
- Browser: Playwright 9

Recovery - Error signature detection (popup text, failed downloads, timeouts, missing controls) and
recovery rules (retry, refresh, reopen, fallback to screenshot, escalate).

Memory - SQLite-backed storage of runs, step traces, evidence artifacts (screenshots, OCR text), learned
selectors, and error frequencies (Python's sqlite3 module is DBAPI compatible and commonly used for
prototyping persistent storage). 10

Safety / policy - Allowed/blocked actions, confirmations, credential protections, domain allowlists,
destructive command gating, and audit logging.

LLM layer - Two roles: 1) Command interpreter: natural language  command + parameters + chosen
workflow
2) Planner / edge-case handler: suggests next step when rules fail, interprets new error text, proposes
recovery actions
- Implement via Responses API (stateful, multimodal, tool/function calling). 11
- Use Structured Outputs to produce schema-valid action calls. 5

Why these phases map to real constraints

        À Desktop automation is most stable when you first identify the correct accessibility/backend
          technology (pywinauto explicitly calls this out as the first step). 12

        À Event subscriptions can reduce wasteful polling on Windows using UI Automation events
          (performance win for always-on observers). 13

        À For the LLM loop, OpenAI's function calling flow is explicitly iterative: request  tool call  execute
           pass tool output  final response (or more calls). 14

        À The Responses API is designed for stateful, multimodal interactions, and supporting tools; it is a
          better fit than "single prompt" approaches for long-running automation. 15

Browser automation with Playwright

Playwright provides installation steps (including browser binaries) and both sync/async APIs. 28 It also
supports browser contexts for isolation and can use nonpersistent contexts that don't write browsing
data to disk (useful for privacy and test-like repeatability). 29

OCR and vision

        À Tesseract: open-source OCR engine (Apache 2.0), includes user manual and training references; no
          built-in GUI. 16

        À OpenCV: broad CV toolkit; modern versions are Apache 2.0 licensed. 30
        À OpenCV preprocessing that commonly improves OCR includes thresholding and morphological

          operations. 31
        À Region detection approaches include contours; OpenCV advises threshold/edge detection before

           findContours for better accuracy. 32
Alternatives worth evaluating (not agent frameworks; just OCR/CV): EasyOCR, PaddleOCR, and cloud OCR.
They can outperform Tesseract on certain fonts/layouts but add heavier dependencies and sometimes GPU
considerations (recommend using them only once you have clear OCR benchmarks for your documents).

GUI frameworks

        À PySide6 (Qt for Python): official Qt bindings; good for richer desktop UI, docking layouts, workflow
          editors, and dashboards. 33

        À Tkinter: standard Python interface to Tcl/Tk; typically bundled and quick to ship small UIs. 34
Recommendation: start with Tkinter for v1 speed, migrate to PySide6 when you need a workflow editor,
multiple panes, and a more scalable UI architecture.

LLM integration patterns

Use the Responses API as the core LLM interface because it supports text+image inputs, stateful
interactions, and tools/function calling. 35

Tool/function calling loop (canonical pattern): request with available tools  receive tool call  execute
tool  send tool output back  final response or more calls. 14

Structured Outputs: enforce JSON Schema compliance when the model outputs actions or structured data
(reduces brittle parsing). 5

Optional (advanced): OpenAI's "computer use" tool guidance highlights harness shapes (built-in tool vs
custom tools layered on top of existing automation harnesses) and recommends isolation and human-in-
the-loop for high-impact actions. Even if you don't use the built-in computer tool, its harness and safety
guidance is directly applicable to your custom executor. 6

Design patterns with schemas and examples

This section provides concrete patterns to keep your platform "engineered," not ad-hoc.

Command  workflow  actions layering

Commands are user-facing (stable names, parameters, permissions).
Workflows are business processes (sequenced steps with validations and branches).
Actions are atomic executor operations (click, type, select window, OCR region, move file).

This layering keeps your system maintainable and makes "teaching" new office processes additive.

Vision and document intelligence pipeline

Why vision needs a pipeline (not just "OCR everything")

Your platform should treat image/document processing as a data extraction pipeline with: 1)
preprocessing
2) region detection (where on the image the field is)
3) OCR
4) parsing
5) confidence scoring
6) validation rules
7) human-in-the-loop review when confidence is low
8) evidence storage (so you can improve over time)

OpenCV offers core preprocessing operations such as thresholding and morphological transforms that are
commonly used prior to OCR. 31

Tesseract can be used via API to extract printed text from images and supports training workflows (useful
when your documents use special fonts/layouts). 16

Recommended pipeline stages

Preprocessing (OpenCV) - Convert to grayscale - Denoise (mild blur) - Binarize (global/adaptive
thresholding) - Morphology (opening/closing) to clean blobs - Resize to improve OCR for small fonts

OpenCV tutorials cover thresholding and morphological operations and the basic functions used. 31

Region detection - Template-based ROI (fastest for stable forms): predefine bounding boxes per document
type - Contour-based detection for boxes/labels: OpenCV contour methods typically work best on binary
images after thresholding. 32 - Hybrid approach: use anchor text ("Invoice No") to locate ROI (advanced)

OCR - Use Tesseract with a configuration tuned to your use case (digits-only, single-line, whitelist characters)
- Keep raw OCR output + confidence-like features (see below) for evidence memory

Confidence scoring Tesseract provides numeric outputs depending on wrapper/tooling; if your OCR
wrapper doesn't expose confidence, use proxies: - did the text match a regex? - did it contain expected
anchor labels? - character set sanity checks (digits only) - cross-field consistency (e.g., totals)

Human-in-the-loop review If confidence < threshold: - UI shows the cropped ROI and proposed value -
user approves/corrects - correction is stored as training/evidence data

                                                                           12
Teaching-by-examples (retraining strategy) - Store (image, ROI box, OCR text, corrected text, doc type,
timestamp) - Track failure clusters by hash of layout + error type - Periodically add new templates/ROIs for
new formats - Consider Tesseract training only when you have enough repeated failures tied to fonts/
layouts (Tesseract explicitly documents training under its manual). 39

OpenCV preprocessing + Tesseract OCR (ROI-based)
(illustrative--production code should add error handling, logging, and evidence saving)

   import cv2
   import pytesseract

   def ocr_roi(image_path: str, roi: tuple[int, int, int, int]) -> str:
          """
          roi = (x, y, w, h)
          """
          img = cv2.imread(image_path)
          if img is None:
                 raise FileNotFoundError(image_path)

          x, y, w, h = roi
          crop = img[y:y+h, x:x+w]

          gray = cv2.cvtColor(crop, cv2.COLOR_BGR2GRAY)
          # Adaptive thresholding is often useful for uneven lighting
          thr = cv2.adaptiveThreshold(

                 gray, 255,
                 cv2.ADAPTIVE_THRESH_GAUSSIAN_C,
                 cv2.THRESH_BINARY,
                 31, 10
          )

          # Morphological opening can reduce small noise
          kernel = cv2.getStructuringElement(cv2.MORPH_RECT, (2, 2))
          clean = cv2.morphologyEx(thr, cv2.MORPH_OPEN, kernel)

          config = r'--oem 1 --psm 7 -c
   tessedit_char_whitelist=0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ-'

          text = pytesseract.image_to_string(clean, config=config).strip()
          return text

Safety, privacy, performance, UX, and quality engineering

Safety and privacy model

A desktop automation platform is effectively "local RPA with superpowers." You need guardrails equivalent
to a professional RPA product.

Permission model - Allowlist by default: only automate specified apps/windows and specified domains -
Action risk levels: low (read-only), medium (data entry), high (send/delete/payment/system settings) - User
confirmations for high-impact actions (delete files, submit forms, send emails)

Credential handling - Prefer OS keychain storage via keyring (system keyring services across platforms).

  20

- On Windows, DPAPI can encrypt secrets so typically only the same user (and often the same machine) can
decrypt. 21
- Never store credentials in plaintext logs or screenshots; automatically redact known input fields and mask
values at logging boundaries.

Logging and audit - Log: command name, workflow id/version, step ids, timestamps, app/window
identifiers, and artifact references (not raw credentials) - Store evidence (screenshots, cropped ROIs, OCR
text) in a structured folder or SQLite blobs, with retention policies

Privacy-by-design tactics - Selective screenshots: only capture when validations fail or when an image-
based step is required - Non-persistent browser contexts for sensitive tasks can avoid writing browsing
data to disk. 29

                                        14
LLM safety - Treat screen/web content as untrusted; require humanintheloop for highimpact actions
(aligned with OpenAI's computer use guidance). 6
- Use Structured Outputs to constrain the model to your action schema. 5

Performance and scalability strategies

Event-driven observation - On Windows, subscribing to UI Automation events can eliminate constant
polling and improve efficiency. 13

Selective capture - Capture screenshots only when: - UI tree is unavailable - validation fails - a new/
unrecognized popup appears - the workflow requires OCR/image extraction

Caching - Cache: - window handles/control identifiers - workflow "stable selectors" - OCR templates/ROIs
per document type - Invalidate caches on structure-changed events (Windows UI Automation structure
changed handlers exist and are documented). 41

Async execution - Use asyncio for IO-bound tasks (browser steps, file downloads, API calls) and for
scheduling multiple workflows safely. 23
- Keep a "single writer" policy for UI automation per desktop session to avoid race conditions; run
parallelism at the workflow queue level, not simultaneous mouse control.

Parallel task handling - Safe parallelism patterns: - Run multiple browser contexts in parallel (Playwright
contexts isolate sessions). 29 - Run background OCR processing while the main UI automation is waiting
on downloads - Avoid concurrent desktop mouse/keyboard actions; use a global executor lock

UX recommendations for the command-based desktop app

Minimal v1 UI - Command entry box + "Run" button - Workflow list (searchable) - Live run log pane - Stop /
Pause / Resume buttons - "Safe mode" toggle (forces confirmations for medium/high risk)

v2+ UI enhancements - Workflow editor: steps as a list with validations and recovery rules - Run
dashboard: last runs, success rates, time saved - Exception inbox: grouped failures with screenshots and
"resume from here" - Evidence review panel for OCR: show ROI + extracted value + approve/correct

Implementation choices: - Tkinter is bundled and fast for simple UI creation. 42
- PySide6 (Qt for Python) is the official Qt binding and scales better for complex editors and dashboards.

Testing, QA, and monitoring

Unit tests - Validate schema parsing: command registry and workflow definitions - Safety policy: allowed/
blocked actions and confirmation gating - Recovery: error signature matching logic

Integration tests - "Dry run" executor with mocked controllers - Playwright sandbox tests using browser
contexts for isolation (Playwright emphasizes isolation via contexts). 46 - Desktop automation tests where
possible; for UI-driven components, focus on smoke tests and replayable scenarios

Failure injection - Simulate: - timeouts - missing controls/selectors - popup interruptions - network offline
(browser) - file lock conflicts - Ensure every failure results in either: - recovery attempted + logged, or - safe
escalation to user with actionable evidence

Metrics to track - Workflow success rate, step success rate - Mean time per workflow - Mean recovery
attempts per workflow - % failures auto-recovered - Exception inbox backlog - Time saved per day/week
(self-reported + measured)
